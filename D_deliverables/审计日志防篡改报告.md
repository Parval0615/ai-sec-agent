# 审计日志防篡改报告

> 日期: 2026-05-07
> 模块: Phase 4.3 — 审计日志哈希链 + 结构化JSON
> 核心升级: 纯文本追加 → JSON结构化 + SHA256哈希链 + 完整性校验

---

## 1. 问题定义

### 1.1 旧版审计日志的问题

升级前 (`security/audit_log.py` v1)：

```
2026-04-24 16:15:55 | cli_user | user | 权限拒绝 | low | 帮我检测... | 权限不足...
```

**三个致命缺陷**：

1. **无完整性校验** — 任何人修改任意一行，系统无法发现
2. **非结构化** — 管道分隔的纯文本，机器难解析，字段截断（输入50字符/结果100字符）
3. **无关联性** — 每条日志独立，不知道前后关系，无法检测"删除中间几行"

### 1.2 攻击场景

```
攻击者执行了危险操作 → 审计日志记录了该操作
  → 攻击者获得文件系统访问权限
  → 修改 audit.log，删除或篡改该条日志
  → 安全审计时无法发现攻击痕迹
```

**防护目标**：修改任意一条日志后，校验函数能精确检测到被篡改的条目。

---

## 2. 哈希链设计

### 2.1 数据结构

每条日志是一个 JSON 对象（一行 = 一条记录）：

```json
{
  "ts": "2026-05-07T14:30:00",
  "user": "admin",
  "role": "admin",
  "op": "策略拦截",
  "risk": "critical",
  "input": "db_query: {'sql': 'DROP TABLE users'}",
  "result": "blocked_sql_keywords: DROP",
  "idx": 42,
  "prev": "abc123def456...",
  "hash": "789012abcde..."
}
```

### 2.2 哈希计算

```
entry[N].hash = SHA256(
    JSON(entry[N] without hash field) + "|" + entry[N-1].hash
)

entry[0].hash = SHA256(
    JSON(entry[0] without hash field) + "|" + "GENESIS_AUDIT_LOG_CHAIN_V2"
)
```

### 2.3 安全保证

| 攻击方式 | 能否检测 | 原理 |
|------|:---:|------|
| 修改一条日志的内容 | 检测 | 该条 hash 不匹配 |
| 删除一条日志 | 检测 | prev_hash 链断裂 |
| 插入一条伪造日志 | 检测 | prev_hash 不匹配前一条 |
| 批量替换从第N条开始的所有日志 | 检测 | 第 N 条 prev_hash 不匹配 |
| 替换整条链（从头构造） | 部分检测 | 最后一条 hash 不同（如果已知正确的 last_hash） |

**无法检测**：完整替换全部日志 + 重新计算所有哈希（如果攻击者有完整文件写入权限且系统无法对比外部备份的 last_hash）。

### 2.4 篡改检测验证结果

实测：修改条目 #1 的 `op` 字段（"测试2" → "被篡改的操作"），不更新 hash。

```
Integrity after tamper: valid=False
Tampered at: [1]
First tampered: #1
条目 #1 被篡改：哈希不匹配
```

校验函数精确检测到被篡改的条目索引。

## 3. 结构化升级

### 3.1 字段对比

| 字段 | 旧版（管道分隔） | 新版（JSON） |
|------|------|------|
| 时间戳 | `2026-04-24 16:15:55` | `"ts": "2026-05-07 14:30:00"` |
| 用户ID | 第2列 | `"user": "admin"` |
| 角色 | 第3列 | `"role": "admin"` |
| 操作类型 | 第4列 | `"op": "策略拦截"` |
| 风险等级 | 第5列 | `"risk": "critical"` |
| 输入内容 | 第6列(截断50字符) | `"input": "..."` (200字符) |
| 结果 | 第7列(截断100字符) | `"result": "..."` (200字符) |
| 索引 | — | `"idx": 42` |
| 前驱哈希 | — | `"prev": "abc123..."` |
| 本条哈希 | — | `"hash": "def456..."` |

### 3.2 向后兼容

- `write_audit_log()` 签名完全不变——所有现有调用无需修改
- `read_audit_log()` 自动识别 JSON 行和旧版管道行，统一展示
- `verify_audit_integrity()` 跳过旧版管道行，仅校验 JSON 条目
- 新增 `read_audit_log_json()` 用于程序化解析

## 5. 局限性分析

### 5.1 局限性1：全链替换无法本地检测

**场景**：攻击者获得文件系统写权限，删除全部旧日志，重新构造一条完整的新链（从头计算所有哈希）。

**根因**：哈希链只能验证**内部一致性**，不能验证**链本身的真实性**。需要外部锚定（如将最新的 `last_hash` 定期写入只读存储或区块链）。

**缓解**：定期将 `last_hash` 写入外部不可变存储（如只读 S3 bucket、区块链、或打印到物理日志）。

### 5.2 局限性2：旧版日志不受保护

**场景**：系统中有 3000+ 条旧版管道分隔日志（v1 格式），这些旧日志没有哈希保护。

**影响**：`verify_audit_integrity()` 跳过旧版日志行。攻击者可以修改旧日志而不被检测。

**缓解**：运行迁移脚本，将旧版日志转为 JSON 格式并计算哈希链。但这会改变旧日志的时间戳和原始格式——需要权衡。

### 5.3 局限性3：截断攻击

**场景**：攻击者删除日志文件的最后 N 行（最新的 N 条记录）。

**当前行为**：哈希链的完整性不受影响（内部一致性保持），但最新操作记录丢失了。

**缓解**：在外部锚定 `last_hash` 的同时也锚定 `last_idx`。当 `last_idx` 小于预期的计数值时，说明有日志被截断。

## 6. 面试速查卡

### 30秒回答："审计日志怎么防篡改？"

"我从纯文本升级到了 JSON 结构化 + SHA256 哈希链。每条日志包含上一条的 SHA256 哈希，形成一条链。`verify_audit_integrity()` 重算所有哈希——修改任意一条日志，该条及之后所有条目的哈希都不匹配，能精确指出被篡改的条目索引。已验证：修改一条日志后，检测函数精确返回 `tampered_at=[1]`。"

### 追问第一层：哈希链能防住什么、防不住什么？

"能防：修改/删除/插入单条或连续多条日志。防不住：全链替换（攻击者删除全部日志重新构造新链）。防全链替换需要外部锚定——定期将 last_hash 写入只读存储或区块链。当前实现是演示级的，展示了哈希链的核心原理。"

### 追问第二层：为什么选 JSON 而不是二进制？

"JSON 人类可读（审计日志的核心需求），机器可解析（支持程序化分析），与现有 Python 生态无缝集成。二进制格式（如 Protocol Buffers）更紧凑但失去了可读性——审计日志需要审计人员能直接看。"

### 追问第三层：旧版日志怎么处理？

"`verify_audit_integrity()` 跳过旧版管道分隔行——它们不受哈希链保护。生产环境需要做一次性迁移：解析旧日志 → 转为 JSON → 按时间顺序重算哈希链。但迁移会改变旧日志的格式和存储时间戳，这是一个需要审慎考虑的取舍。"

---

## 7. 改进路线

1. **外部锚定**: 定期将 `last_hash + last_idx` 写入只读外部存储
2. **日志迁移**: 将旧版管道分隔日志转为 JSON 格式并入哈希链
3. **日志轮转**: 按日期/大小自动创建新日志文件，每个文件独立哈希链
4. **实时告警**: 在 `verify_audit_integrity()` 检测到篡改时主动通知管理员
