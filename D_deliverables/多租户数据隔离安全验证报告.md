# 多租户数据隔离安全验证报告

> 日期: 2026-05-07
> 测试范围: Chroma collection_name 隔离 + LangGraph checkpointer thread_id 隔离
> 测试用例: 5项（2 通过 / 1 警告 / 2 通过含已知风险）

---

## 1. 问题定义

### 1.1 威胁模型

在多租户 SaaS 场景中，多个租户共享同一个 RAG Agent 实例：

```
租户A ──→ 上传文档 ──→ RAG 向量库 (collection_A) ──→ 检索回答
租户B ──→ 上传文档 ──→ RAG 向量库 (collection_B) ──→ 检索回答
```

**核心安全需求**：租户A的文档内容、对话历史绝不应被租户B获取。

### 1.2 双层隔离架构

```
┌─────────────────────────────────────────────┐
│              应用层 (graph_agent)              │
│  thread_id → 对话隔离 (LangGraph Checkpointer) │
│  user_id   → 审计追踪                         │
└─────────────────────────────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│            数据层 (core/rag)                   │
│  session_id → collection_name → 向量隔离      │
│  ChromaDB PersistentClient 持久化             │
└─────────────────────────────────────────────┘
```

| 隔离层 | 隔离键 | 隔离粒度 | 存储 |
|------|------|------|------|
| 对话历史 | `thread_id` | 每次对话 | SQLite (`checkpoints/graph_state.db`) |
| 文档数据 | `session_id` → `collection_name` | 每个租户+每次会话 | ChromaDB (`chroma_db/`) |

## 2. 隔离测试

### 2.1 Test 1: Chroma Collection 隔离

**测试方法**：租户A上传含恶意指令的 PDF → 租户B用相同 PDF 上传 → 租户B检索租户A的恶意内容

**结果: PASS**

| 指标 | 值 |
|------|------|
| 租户A的 collection_name | `rag_docs_isolation_tenant_a` |
| 租户B的 collection_name | `rag_docs_isolation_tenant_b` |
| Collection 名称是否不同 | 是 |
| 租户B检索到租户A的数据 | 否 |

**结论**：`session_id` 参数编码到 `collection_name`，不同 session_id 产生不同的 Chroma collection，数据物理隔离。

### 2.2 Test 2: Checkpointer Thread ID 隔离

**测试方法**：租户A通过 `thread_id="tenant_a"` 存入敏感数据 → 租户B通过 `thread_id="tenant_b"` 读取对话历史

**结果: PASS**

| 指标 | 值 |
|------|------|
| 租户A能看到自己的数据 | 是 |
| 租户B能看到租户A的数据 | 否 |
| 租户A的消息数 | >0 |
| 租户B的消息数 | 0（未创建历史） |

**结论**：`thread_id` 是 LangGraph Checkpointer 的隔离键。不同 `thread_id` 产生完全独立的对话历史。

### 2.3 Test 3: Thread ID 碰撞

**测试方法**：两个不同的"用户"使用**相同** `thread_id` 进行对话

**结果: WARNING（已知设计选择）**

| 指标 | 值 |
|------|------|
| thread_id 被共享 | 是 |
| 用户A的数据被用户B可见 | 是 |

**分析**：`thread_id` 设计为**对话标识符**，而非安全边界。当两个用户共享 `thread_id`，他们看到同一个对话——这在协作场景中是预期行为，但在多租户场景中是安全风险。

**缓解措施**：应用层必须保证 `thread_id` 的唯一性——通过 `user_id + uuid` 组合生成，或使用 HMAC 签名的 `thread_id`。

### 2.4 Test 4: SQLite 直接访问

**测试方法**：绕过 graph_agent API，直接读取 `checkpoints/graph_state.db`

**结果: PASS（已知风险已记录）**

| 指标 | 值 |
|------|------|
| 数据库文件可读 | 是 |
| 可直接读取所有 thread_id | 是 |
| 可直接读取所有对话内容 | 是 |

**风险级别**：中。需要文件系统访问权限。但在共享主机/容器环境中，如果未正确配置文件系统权限，攻击者可能通过路径遍历或其他漏洞读取 SQLite 文件。

**缓解措施**：文件系统权限控制（`chmod 600`）、定期审计 checkpoints 文件访问、生产环境迁移到需要认证的数据库（PostgreSQL + 行级安全策略）。

### 2.5 Test 5: ChromaDB 直接访问

**测试方法**：绕过 `core/rag` API，直接用 ChromaDB PersistentClient 读取所有 collection

**结果: PASS（已知风险已记录）**

| 指标 | 值 |
|------|------|
| 总 Collection 数 | 12 |
| Collection 名称唯一 | 是 |
| 可直接查询任意 Collection | 是 |

**风险级别**：中。与 SQLite 相同的文件系统权限问题。ChromaDB PersistentClient 没有内置的认证/授权——任何能访问 `chroma_db/` 目录的进程都能读取所有租户的向量数据。

## 3. 隔离方案的设计假设

### 3.1 假设1：session_id 唯一性

**假设**：应用层为每个租户+每次会话生成唯一的 `session_id`。

**当前实现**：`streamlit_app.py` 用 `session_id = str(uuid.uuid4())` 生成。CLI 模式依赖用户传入。

**绕过路径**：如果攻击者能够预测或猜测另一个租户的 `session_id`，可以构造相同的 `collection_name` 并获取相同的 retriever 对象。UUID4 的碰撞概率极低（~10^-18），但应用层不验证 `session_id` 的所有权——任何知道 `session_id` 的人都能访问对应的 collection。

### 3.2 假设2：thread_id 是对话标识符而非安全边界

**假设**：`thread_id` 用于关联多轮对话，不承载安全语义。安全由 `session_id`（数据层）和文件系统权限（存储层）保证。

**绕过路径**：如果应用层使用可预测的 `thread_id`（如递增整数），攻击者可以遍历 thread_id 读取其他用户的对话历史。

### 3.3 假设3：文件系统安全

**假设**：运行 Agent 的服务器文件系统权限正确配置，`chroma_db/` 和 `checkpoints/` 目录仅对 Agent 进程可读。

**绕过路径**：容器逃逸、路径遍历漏洞、错误配置的共享存储（NFS/S3）、备份文件未加密——任何对文件系统的未授权访问都会绕过两层应用隔离。

## 4. 安全加固建议

### 4.1 application_id 签名（推荐）

对 `session_id` 做 HMAC 签名，阻止攻击者伪造：

```python
import hmac, hashlib

def sign_session_id(tenant_id: str, session_id: str, secret: str) -> str:
    payload = f"{tenant_id}:{session_id}"
    signature = hmac.new(secret.encode(), payload.encode(), hashlib.sha256).hexdigest()[:16]
    return f"{tenant_id}_{session_id}_{signature}"

def verify_session_id(signed: str, secret: str) -> tuple[str, str]:
    parts = signed.rsplit("_", 1)
    tenant, session, sig = parts[0].split("_", 1)[0], parts[0].split("_", 1)[1], parts[1]
    expected = sign_session_id(tenant, session, secret).rsplit("_", 1)[1]
    if not hmac.compare_digest(sig, expected):
        raise PermissionError("Invalid session_id signature")
    return tenant, session
```

### 4.2 thread_id 前綴隔離

在應用層強制 `thread_id` 包含租戶前綴：

```python
def make_thread_id(tenant_id: str, conversation_id: str) -> str:
    return f"tenant_{tenant_id}__conv_{conversation_id}"

# 在 get_thread_messages 中校验
def get_tenant_messages(tenant_id: str, conversation_id: str) -> list:
    tid = make_thread_id(tenant_id, conversation_id)
    return get_thread_messages(tid)
```

### 4.3 数据库升级路径

生产环境建议从 SQLite/ChromaDB PersistentClient 升级：
- **对话历史**: SQLite → PostgreSQL + 行级安全策略（RLS）
- **向量存储**: ChromaDB PersistentClient → ChromaDB Server（带 API 认证）或 pgvector + RLS

## 5. 面试速查卡

### 30秒回答："多租户数据隔离怎么做？"

"两层隔离：对话历史通过 LangGraph Checkpointer 的 thread_id 隔离（SQLite），文档数据通过 Chroma collection_name 隔离（session_id 编码到 collection 名）。5 项测试验证了基本隔离有效。已知两个局限：(1) thread_id 是对话标识符不是安全边界，应用层要保证唯一性，(2) 文件系统权限是最后防线——SQLite 和 ChromaDB 文件如果被直接读取，所有隔离都失效。"

### 追问第一层：thread_id 碰撞怎么办？

"thread_id 碰撞不是 bug 是设计选择——它用于关联多轮对话，不承载安全语义。防碰撞的措施是应用层规则：(1) thread_id 包含租户前缀，(2) 对 thread_id 做 HMAC 签名验证，(3) 生产环境用数据库 RLS 而不是文件系统隔离。"

### 追问第二层：ChromaDB 本身有没有认证？

"PersistentClient 模式没有——它就是本地文件存储。Server 模式支持 API Token 认证。两者的权衡是部署复杂度 vs 安全——PersistentClient 零配置适合演示，Server 模式需要额外的基础设施但支持真正的多租户安全。"

### 追问第三层：你能说出隔离方案的设计假设和可能被绕过的路径吗？

"三个关键假设和对应的绕过路径：(1) session_id 唯一性假设——绕过：预测/猜测 session_id（UUID4 碰撞概率极低但应用层不做所有权验证），缓解：HMAC 签名。(2) thread_id 是对话标识符假设——绕过：遍历可预测的 thread_id 读取其他用户历史，缓解：thread_id 包含租户前缀。(3) 文件系统安全假设——绕过：容器逃逸/路径遍历/共享存储未加密/备份文件泄露，缓解：数据库迁移 + RLS。其中最薄弱的是文件系统假设——因为 SQLite 和 ChromaDB 文件一旦被读，所有应用层隔离都透明。"
