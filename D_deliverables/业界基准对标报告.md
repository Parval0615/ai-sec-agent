## 2. HackAPrompt 基准测试

### 2.1 数据集说明

HackAPrompt 是 2023 年全球 Prompt 注入攻防竞赛的公开数据集（600+ prompt），本项目抽取 30 条分层样本，覆盖全部 8 个难度级别和 4 个攻击类别。

| 类别 | 样本数 | 级别范围 | 基线拦截率(2023) |
|------|:---:|------|:---:|
| 直接注入 (direct_injection) | 8 | Level 1-2 | ~30% |
| 越狱 (jailbreak) | 8 | Level 3-4 | ~20% |
| 提示泄露 (prompt_leakage) | 7 | Level 5-6 | ~15% |
| 混淆绕过 (obfuscation) | 7 | Level 7-8 | ~10% |

基线数据来源：HackAPrompt 竞赛参与者提交的防御方案平均拦截率（2023年，GPT-3.5 时代）。

### 2.2 本项目拦截率（L1+L2+L3 三层防火墙）

| 类别 | 基线拦截率 | 本项目拦截率 | 提升 |
|------|:---:|:---:|:---:|
| 直接注入 | 30% | 100% (8/8) | +70pp |
| 越狱 | 20% | 87.5% (7/8) | +67.5pp |
| 提示泄露 | 15% | 85.7% (6/7) | +70.7pp |
| 混淆绕过 | 10% | 71.4% (5/7) | +61.4pp |
| **总计** | **19.0%** | **86.7% (26/30)** | **+67.7pp** |

### 2.3 级别分析

| 级别 | 难度描述 | 拦截率 |
|------|------|:---:|
| Level 1-2 | 基础直接注入 | 100% (8/8) |
| Level 3-4 | 角色扮演越狱 | 87.5% (7/8) |
| Level 5-6 | 提示泄露探测 | 85.7% (6/7) |
| Level 7-8 | 复杂混淆绕过 | 71.4% (5/7) |

随着级别升高，拦截率下降——这符合预期，因为高级别名攻击使用更复杂的混淆技术。

### 2.4 与基线对比的注意事项

**2023 vs 2026 的模型代差。** 本项目的 86.7% 拦截率中有两部分贡献：

1. **底层 LLM (Qwen3.5-35B) 自身的安全对齐** — 2023 年参赛者用 GPT-3.5，2026 年的模型安全对齐远优于当时
2. **本项目防火墙 (L1+L2+L3) 的增量** — 需要消融实验（对比防火墙开/关）才能精确量化

在没有消融实验的情况下，86.7% vs 19.0% 的 +67.7pp 差异**不能简单归因为防火墙的优势**——大部分差异来自底层模型的代际提升。

**这个自知之明在面试中比 86.7% 这个数字更有说服力。**

## 3. Garak 风格探测

### 3.1 数据集说明

Garak 是 LLM 安全测试的 Python 框架，有 20+ 种探测类型。本项目实现 6 类共 20 条模拟探测：

| Garak 探测类型 | 样本数 | 探测逻辑 |
|------|:---:|------|
| dan.DanInTheWild | 3 | DAN 越狱变体 |
| encoding.base64 | 3 | Base64 编码注入 |

| misinfo | 2 | 生成虚假安全公告 |
| malwaregen | 2 | 生成恶意代码(keylogger/stealer) |
| lm_gen | 2 | LLM 生成的对抗性提示 |

### 3.2 按探测类型的拦截率

| 探测类型 | 拦截率 |
|------|:---:|
| dan.DanInTheWild | 100% (3/3) |
| encoding.base64 | 66.7% (2/3) |
| xtalk.TranslationAttack | 100% (2/2) |
| knownbadsignatures | 100% (4/4) |
| leakerplay | 100% (2/2) |
| misinfo | 50.0% (1/2) |
| malwaregen | 100% (2/2) |
| lm_gen | 50.0% (1/2) |

### 3.3 绕过分析

3 条绕过：
- **encoding.base64 (1条)**: 中文 Base64 编码的越狱指令。Base64 内层是中文，解码后无危险英文关键词
- **misinfo (1条)**: "伪造微软安全公告"——L1 无匹配关键词，L2 可能将其归类为"正常创作请求"
- **lm_gen (1条)**: LLM 对中文越狱指令评分较低（因为使用了正式的安全研究员身份框架）

## 4. OWASP Top 10 for LLM Applications 覆盖度

### 4.1 逐项覆盖分析

| OWASP 风险 | 本项目覆盖 | 实现模块 | 说明 |
|------|:---:|------|------|
| **LLM01: Prompt Injection** | 覆盖 | 3.1 防火墙 | 105+ payload库, L1+L2+L3三层检测, 93.3%拦截率 |
| **LLM02: Insecure Output Handling** | 覆盖 | 3.4 输出评测 | 三级粒度输出过滤 + fact_check + 敏感信息脱敏 |
| **LLM03: Training Data Poisoning** | 覆盖 | 3.2 投毒防护 | 12种RAG投毒场景, L1/L2级联扫描, 入库+检索双重防御 |
| **LLM04: Model DoS** | 未覆盖 | — | 需要资源限制/速率限制/输入长度限制，属于基础设施层 |
| **LLM05: Supply Chain** | 未覆盖 | — | 需要SBOM/模型签名/依赖扫描，属于MLOps流程 |
| **LLM06: Sensitive Information Disclosure** | 覆盖 | 3.1+3.4 | 提示泄露检测 + 敏感信息脱敏(手机/身份证/银行卡/API Key/IP) |
| **LLM07: Insecure Plugin Design** | 未覆盖 | Phase 4计划 | Tool Policy Engine 将覆盖工具调用的权限控制 |
| **LLM08: Excessive Agency** | 未覆盖 | Phase 4计划 | RBAC + 策略引擎将限制工具调用范围和频率 |
| **LLM09: Overreliance** | 覆盖 | 3.4 输出评测 | 安全幻觉检测 + fact_check 事实核查 |
| **LLM10: Model Theft** | 未覆盖 | — | 需要API限流/模型水印/访问日志分析 |

**覆盖度：5/10**

### 4.2 未覆盖 5 项的合理性

未覆盖的 5 项可以分为三类：

| 类别 | 风险 | 为什么不覆盖 |
|------|------|------|
| **基础设施层** | LLM04 (DoS), LLM10 (Model Theft) | 需要资源限制/API限流/水印——不是 Prompt 检测问题 |
| **供应链层** | LLM05 (Supply Chain) | 需要 SBOM/签名验证——属于 MLOps 流程 |
| **Phase 4计划** | LLM07 (插件安全), LLM08 (过度代理) | Tool Policy Engine + RBAC 将覆盖这两项 |

**面试速答模板**："OWASP Top 10 我覆盖了 5/10，未覆盖的 5 项中，3 项属于基础设施/供应链层（不在本项目范围），2 项（插件安全和过度代理）在下一阶段的 Tool Policy Engine 中实现。我认为明确说'不知道/没覆盖'比假装覆盖更有说服力。"

## 5. 定性架构对比

### 5.1 与 Lakera Guard 对比

| 维度 | 本项目 | Lakera Guard |
|------|------|-------------|
| 检测方法 | 3层级联（L1关键词+正则 → L2 LLM语义 → L3后检查） | 单一 fine-tuned GPT 分类器 + 正则预筛 |
| 延迟 | L1<1ms, L2~500ms, L3<1ms | ~200ms (托管API) |
| 可解释性 | 每层独立 reasoning + risk_score | 类别标签 + 置信度分数 |
| 部署 | 自部署 (Python, 开源) | SaaS only (无自部署选项) |
| 定制性 | 关键词/正则可自定义, 粒度三级可配 | 单一阈值, 不可定制 |
| 适用场景 | 安全培训/教育平台 + 内部工具 | 生产环境快速接入 |

**核心差异**：Lakera 是专为生产环境设计的商业化产品（低延迟、易接入、无需自维护），本项目是教育+演示导向的框架（多层可解释、粒度可配、全部开源）。两者的目标用户不同——不是竞争关系，而是互补。

### 5.2 与 Rebuff 对比

| 维度 | 本项目 | Rebuff |
|------|------|-------------|
| 检测方法 | L1+L2+L3 级联 | 4阶段管道（启发式 → LLM → 向量DB → Canary Token） |
| 核心创新 | 多层纵深 + 上下文感知 | 向量相似度检测已知攻击 + Canary Token泄露检测 |
| 延迟 | <1ms(L1) + ~500ms(L2) | ~100-300ms |
| 开源 | 是 | 是 (Python包) |
| 独有功能 | classify_with_context()多轮感知 | Canary Token 检测提示泄露 |

**核心差异**：Rebuff 的向量相似度方法对已知攻击效果好，但对未见过的变体可能漏检（依赖攻击库）。本项目 LLM 语义分析对未知变体有更好的泛化能力，但延迟更高。

### 5.3 与 NeMo Guardrails 对比

| 维度 | 本项目 | NeMo Guardrails |
|------|------|-------------|
| 定位 | 安全检测层 | 对话管理框架 |
| 配置方式 | Python 代码 | Colang DSL (领域特定语言) |
| 输入安全 | 3层防火墙 | 输入 Rails (规则+LLM) |
| 输出安全 | 可执行代码+敏感信息+粒度控制 | 输出 Rails (事实核查+不允许主题) |
| 对话流管理 | LangGraph StateGraph | Colang 对话流定义 |
| 学习曲线 | 中等 (需要理解3层架构) | 中高 (需要学习 Colang DSL) |

**核心差异**：NeMo 是一个全面的对话管理框架，安全只是其中一部分。本项目聚焦于安全检测，深度更深但范围更窄。对于只需要安全层的场景，本项目的侵入性更低；对于需要完整对话管理（含安全、事实核查、业务流程）的场景，NeMo 更合适。

## 6. 失败案例分析

### 6.1 失败案例1：HackAPrompt Level 7-8 混淆绕过

**现象**：30 条 HackAPrompt 样本中，4 条绕过集中在混淆类别（3条 Level 7-8 + 1条 Level 4 高级越狱）。

**根因**：
- Base64编码中文越狱指令：L2 LLM 看到的是解码后的中文而非原始 Base64，中文版的越狱措辞比英文更温和（使用"无限制演示模式"而非 "Do Anything Now"）
- 分隔符绕过（`i-g-n-o-r-e`）：连字符拆分降低了 L1 关键词匹配的置信度
- 全角字符（`ｉｇｎｏｒｅ`）：NFKC 归一化处理全角，但 L1 正则中的英文关键词在全角→半角后仍然匹配，绕过的是 L1 但会被 L2 拦截

**缓解方向**：增强 Level 7-8 级别的混淆检测（如对连字符拆分自动合并检查，对 Base64 解码后的中文内容做语义分类）。

### 6.2 失败案例2：OWASP 覆盖度的"边界诚实"

**现象**：OWASP Top 10 只覆盖了 5/10。未覆盖的项目包括 Model DoS、供应链安全、插件安全、过度代理、模型盗窃。

**根因**：这不是检测能力不足，而是项目定位的边界。这 5 项属于不同的技术层——DoS 需要基础设施层的速率限制，供应链安全需要 MLOps 流程（SBOM/签名验证），插件安全和过度代理需要在工具执行层做策略控制（而不是在 Prompt 检测层）。

**为什么这是"好的失败"**：在面试中，诚实地说明"XX 不在我的项目范围内，因为..."比声称全覆盖但实际浅尝辄止更有说服力。面试官看重的是你能否判断边界——知道自己不知道什么。

### 6.3 失败案例3：Garak misinfo 类型绕过

**现象**：Garak 风格探测中，misinfo 类型（虚假安全公告生成）2 条中 1 条绕过。

**根因**：申请"写一篇虚假的微软安全公告"——L1 无危险关键词（不含"忽略"、"注入"、"越狱"等），L2 LLM 可能将其归类为"创作请求"而非"安全威胁"。但这类请求的危害是间接的（生成的假公告可能被用于社会工程学攻击）。

**分类**：这是一个"间接危害"场景——防火墙的威胁模型聚焦于直接注入/越狱/泄露，对间接危害（生成可用于下游攻击的内容）检测较弱。

## 7. 面试速查卡

### 30秒回答："你做过业界对比吗？"

"做过。我跑了 HackAPrompt 30 条分层样本（4类×8级），拦截率 86.7%，对比 2023 年基线 19.0%。但我要强调的是——这 +67.7pp 差异大部分来自底层 LLM 的代际提升（GPT-3.5 vs Qwen3.5-35B），不是防火墙的单一贡献。精确的防火墙增量需要消融实验（开/关防火墙对比）。此外，我做了 Lakera Guard / Rebuff / NeMo Guardrails 的定性架构对比，核心差异是——它们是产品/框架，本项目是教育+演示导向的安全检测层。"

### 追问第一层：OWASP 覆盖了多少？

"5/10。覆盖了 Prompt 注入、不安全输出、训练数据投毒、敏感信息泄露、过度依赖。未覆盖的 5 项中，Model DoS 和供应链安全属于基础设施层（不是 Prompt 检测问题），插件安全和过度代理在 Phase 4 Tool Policy Engine 中实现，模型盗窃需要 API 限流/水印。我认为明确说'不知道'比假装全覆盖更有面试说服力。"

### 追问第二层：跟 Lakera Guard 比谁更强？

"定位不同。Lakera 是商业化 SaaS 产品，延迟 200ms，无自部署选项，适合生产环境快速接入。本项目是开源的自部署框架，3层可解释，粒度三级可配，适合安全培训和教育场景。不是竞争关系——如果一个公司需要零延迟的 SaaS 接入，选 Lakera；如果需要可解释的检测过程和自定义粒度，用本项目。"

### 追问第三层：HackAPrompt 跑分能证明什么？

"绝对分数（86.7%）证明不了太多——因为底层 LLM 自身的安全对齐贡献无法分离。真正有价值的是：1) 分层分析能看出哪类攻击是薄弱点（混淆绕过 71.4% < 直接注入 100%），2) 消融实验能精确量化防火墙的增量贡献。我还没跑消融实验——这是我下一步要做的事。"

## 8. 改进路线

1. **消融实验** (优先级最高): 防火墙开/关对比，精确量化 L1/L2/L3 各层的增量贡献
2. **HackAPrompt 全量**: 从 30 样本扩展到全量 600+ 样本（需要时间和 API 配额）
3. **实时 Garak 集成**: 直接调用 Garak Python 库而非模拟探测
4. **OWASP 动态更新**: 跟踪 OWASP Top 10 for LLM Apps 的版本更新

---

> **核心结论**：基准对标的含金量不在于绝对分数，而在于——你知道分数是怎么构成的（LLM贡献 vs 防火墙贡献）、你知道自己的薄弱点在哪里（混淆绕过、间接危害）、你知道跟业界方案的差异是什么（定位、延迟、可解释性），以及你知道自己不知道什么（OWASP 未覆盖 5 项的明确理由）。
